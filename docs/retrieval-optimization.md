# 检索优化模块

## 核心架构

```
用户查询 → 查询路由(list/detail) → 提取过滤条件（仅list） → 带过滤的混合检索 → RRF重排 → 返回结果
```

**关键设计：Detail查询不过滤**
- **List查询**（如"推荐几个简单的素菜"）：提取过滤条件 → `metadata_filtered_search`
- **Detail查询**（如"宫保鸡丁怎么做"）：直接 `hybrid_search`，不应用过滤条件
- **原因**：用户已明确指定菜名，过滤条件会干扰检索结果（如误匹配同义词导致过滤掉正确结果）

## 三层过滤条件提取

| 层级 | 匹配方式 | 示例 | 覆盖场景 |
|------|---------|------|---------|
| Layer 1 | 精确匹配 | "素菜"→分类=素菜 | 标准分类名/难度名 |
| Layer 2 | 同义词扩展 | "素食"→分类=素菜 | 同义词/变体词 |
| Layer 3 | 场景词映射 | "新手"→难度∈[非常简单,简单] | 场景/需求描述 |

### 同义词表（CATEGORY_SYNONYMS）
- "素食""青菜""蔬菜" → 素菜
- "海鲜""鱼""虾" → 水产
- "汤""羹""炖品" → 汤品

### 场景词映射表（SCENE_TO_FILTER）
- "新手""快手""简单""入门" → 难度∈[非常简单, 简单]
- "硬菜""复杂""挑战" → 难度∈[困难, 非常困难]
- "健身""减脂" → 分类∈[素菜, 水产], 难度∈[非常简单, 简单, 中等]
- "下酒" → 分类∈[荤菜, 素菜], 难度∈[简单, 中等]

### 菜系映射表（CUISINE_TO_CATEGORY）
- "川菜""湘菜""粤菜" → 分类∈[荤菜, 素菜, 汤品]（多分类OR检索）

### 查询类型感知过滤

| 查询类型 | 过滤策略 | 原因 |
|---------|---------|------|
| **List** | 提取过滤条件，应用 `metadata_filtered_search` | 用户需要筛选列表，过滤条件缩小候选范围 |
| **Detail** | 跳过过滤，直接使用 `hybrid_search` | 用户已明确指定菜名，过滤会干扰精确匹配 |
| **General** | 有过滤条件则应用，否则直接混合检索 | 保持灵活性 |

实现代码（`main.py`）：
```python
if route_type == 'detail':
    # Detail查询跳过过滤，避免误过滤正确结果
    relevant_chunks = self.retrieval_module.hybrid_search(rewritten_query, top_k=...)
elif filters:
    relevant_chunks = self.retrieval_module.metadata_filtered_search(
        rewritten_query, filters, top_k=...
    )
else:
    relevant_chunks = self.retrieval_module.hybrid_search(rewritten_query, top_k=...)
```

## 检索策略

### 向量检索（`_filtered_vector_search`）

- FAISS原生不支持元数据过滤，采用"后过滤"策略
- **关键参数：** `fetch_k = min(总文档数, max(200, k * 5))`
  - 1977个chunk时，k=5则fetch_k=200
  - 先检索200个候选，再应用filter，确保过滤后有足够结果
- 无过滤条件时，直接使用标准FAISS检索

### BM25检索（`_filtered_bm25_search`）

- 先过滤候选集，再在候选集内检索
- **优化：** 候选集≤100时，直接用关键词匹配排序（避免重建BM25索引开销）
- 候选集>100时，重建BM25索引后检索

### 多值OR检索（`_multi_value_search`）

**问题：** 用户说"川菜"，但川菜包含荤菜、素菜、汤品等多个分类，传统单值过滤无法处理

**解决方案：** 统一使用列表值表示OR条件

```python
# 多分类OR检索
filters = {'category': ['荤菜', '素菜', '汤品']}

# 多难度OR检索
filters = {'difficulty': ['非常简单', '简单']}
```

**实现细节：**
- `_filter_documents` 统一处理单值和列表值（列表表示OR）
- `_multi_value_search` 统一处理所有字段的多值OR检索
- 候选集不大时直接返回，避免不必要的语义检索开销

**智能候选集策略：**

```
候选集大小 ≤ 200个chunks → 直接返回候选集（List查询的核心是满足过滤条件）
候选集大小 > 200个chunks → 对每个值分别检索，合并去重后RRF重排
```

**原因：** List查询的核心需求是"满足过滤条件"，而非"与查询文本语义最相关"。例如"推荐适合新手的菜"，用户就是要"简单菜"的列表，不需要这些菜与"推荐"这个词高度相关。

## RRF重排

综合向量检索和BM25检索的排名信息：

```python
score = 1.0 / (60 + rank + 1)  # k=60
```

## 迭代记录

### 2026.04.28 Detail查询过滤修复

**问题背景：**
Detail查询（如"宫保鸡丁怎么做"）在应用过滤优化后效果反而变差。

**根因分析：**
- `_extract_filters_from_query` 对Detail查询过度提取过滤条件（如"蛋糕"误匹配"烘焙"→"甜品"分类）
- 过滤后FAISS候选集减少、BM25退化为简单关键词匹配，导致正确结果丢失

**解决方案：**
- Detail查询跳过metadata过滤，直接走 `hybrid_search`
- List查询保持过滤优化逻辑不变

**效果对比（修复后）：**

| 指标 | Baseline | Optimized | 提升 |
|------|----------|-----------|------|
| recall@3 | 0.631 | **0.842** | **+0.211 (+33.4%)** |
| recall@5 | 0.642 | **0.853** | **+0.211 (+32.9%)** |
| recall@10 | 0.653 | **0.864** | **+0.211 (+32.3%)** |
| precision@3 | 0.433 | **0.644** | **+0.211 (+48.7%)** |
| mrr | 0.800 | **0.900** | **+0.100 (+12.5%)** |

**按查询类型细分：**

| 查询类型 | 指标 | Baseline | Optimized | 提升 |
|---------|------|----------|-----------|------|
| **Detail** | recall@3 | 0.839 | 0.839 | +0.000（保持） |
| **Detail** | recall@5 | 0.859 | 0.859 | +0.000（保持） |
| **Detail** | mrr | 0.941 | 0.941 | +0.000（保持） |
| **List** | recall@3 | 0.359 | **0.846** | **+0.487 (+135.7%)** |
| **List** | recall@5 | 0.359 | **0.846** | **+0.487 (+135.7%)** |
| **List** | mrr | 0.615 | **0.846** | **+0.231 (+37.6%)** |
| **List** | filter_accuracy@3 | 0.359 | **0.846** | **+0.487 (+135.7%)** |

**结论：**
- Detail查询：修复后保持baseline水平（不过滤就不下降）
- List查询：过滤优化效果显著，recall@3从0.359提升至0.846
- 整体指标大幅提升，验证"查询类型感知过滤"策略有效

### 2026.04.27 检索系统全面升级

**问题背景：**
1. 过滤后有效结果太少，系统不得不补充未过滤的结果，导致过滤条件形同虚设
2. 语义检索无法理解"新手""快手"等场景词

**解决方案：**
- 过滤条件提取从查询重写之后移到之前，防止关键词丢失
- 三层语义过滤提取（精确匹配→同义词扩展→场景词映射）
- 多值OR检索统一处理

**效果对比（首次优化）：**

| 指标 | Baseline | Optimized | 提升 |
|------|----------|-----------|------|
| recall@3 | 0.476 | 0.758 | +0.282 (+59.2%) |
| recall@5 | 0.487 | 0.758 | +0.271 (+55.6%) |
| recall@10 | 0.498 | 0.758 | +0.260 (+52.2%) |
| precision@3 | 0.278 | 0.424 | +0.146 (+52.5%) |
| mrr | 0.533 | 0.818 | +0.285 (+53.5%) |

**实验设计：**
- **Baseline**：跳过过滤条件提取，直接 `hybrid_search(query)`（模拟优化前的逻辑）
- **Optimized**：提取过滤条件 → `metadata_filtered_search(query, filters)`（当前优化逻辑）

> ⚠️ **注意**：04.27版本的Optimized包含Detail查询误过滤问题，已于04.28修复。
